## 🧬 一、结构层：精密的层级调度体系

这套设计的核心，是将一个复杂任务的理解、规划和执行，分配给不同角色，形成一个清晰的**指挥链（Chain of Command）**。

### 1.1 🔮 元Agent（MetaAgent）：大脑与总调度

**定位**：整个体系的唯一入口和总调度中心。

**核心职责**：
- **意图解析（Intent Resolution）**：理解你的模糊概念或具体指令，澄清歧义。
- **蓝图制定（Blueprinting）**：将抽象目标拆解为可执行的步骤序列。
- **专家调度（Expert Dispatch）**：从“专家库”中挑选最合适的子 Agent 去执行。
- **信任分级（Trust Grading）**：根据用户信任等级，决定哪些操作可以自动执行，哪些需要人工确认。

**设计原则**：
> **单一入口原则**：所有外部请求必须经过 MetaAgent，禁止直接调用子 Agent。
> **委托式编排**：MetaAgent 不亲自干活，只负责“想”和“派”。

---

### 1.2 👥 子Agent（SubAgents）：专家库

**定位**：负责“干活”的领域专家。我们设计了一个**可插拔的插件体系**，每个 Agent 都在自己的领域内解决特定问题。

#### **代码类专家**
| Agent | 职责 | 典型场景 |
|-------|------|---------|
| `CodeExplainer` | 代码解释 | “这段代码是做什么的？” |
| `CodeGenerator` | 代码生成/重构 | “帮我写一个快速排序算法” |
| `CompletionAgent` | 行内补全 | FIM（Fill-in-Middle）实时补全 |
| `SQLOptimizer` | SQL 优化 | “这个查询太慢了，怎么优化？” |

#### **项目管理专家**
| Agent | 职责 | 典型场景 |
|-------|------|---------|
| `FileSystemAgent` | 文件管理 | 创建、移动、删除文件 |
| `GitAgent` | Git 操作 | 提交、分支管理、冲突解决 |
| `PlannerAgent` | 复杂任务规划 | 多步骤任务的 DAG 分解 |

#### **系统与知识专家**
| Agent | 职责 | 典型场景 |
|-------|------|---------|
| `WebSearcher` | 网络搜索/知识获取 | “最新 React 19 的特性是什么？” |
| `ProactiveEngine` | 主动建议引擎 | “我注意到你经常写这类代码，要不要存个技能？” |

**设计原则**：
> **单一职责原则（SRP）**：每个 Agent 只做一件事，并做好。
> **双向端口原则**：Agent 实现 `SubAgent` 端口（对外提供服务），通过构造函数注入依赖（对内消费服务）。

## ✨ 二、行为层：Think-Act-Observe 自主循环

这是让 Agent 从“被动的工具”变成“主动的学徒”的关键设计。每个子 Agent 在执行任务时，都遵循一个**思考-行动-观察**的标准流程。

### 2.1 🧠 Think（思考）

**过程**：
1. **背景注入**：从记忆系统（MemoryPortal）中检索相关的历史操作、项目上下文、用户偏好。
2. **深度推理**：利用大模型（如 DeepSeek V4 Thinking Mode）进行深度推理。
   - 不只是想“怎么做”，还会思考“为什么这么做”。
   - 评估潜在风险和替代方案。
3. **决策制定**：决定下一步的具体行动（比如调用哪个工具、传递什么参数）。

**关键能力**：
- **缺失信息推断**：如果用户指令模糊，Agent 会基于历史记忆推断用户可能想要的细节。
- **多轮澄清**：如果歧义太大，Agent 会通过 EventBus 发布 `ClarificationRequestEvent`，请求用户澄清。

---

### 2.2 🛠️ Act（行动）

**过程**：
1. **工具调用**：Agent 根据思考结果，通过 `Function Calling` 机制，自主调用外部工具。
2. **权限校验**：所有工具调用经过 `ToolGateway` 进行权限校验和审计。
   - **低风险操作**（如读取文件）：自动执行。
   - **高风险操作**（如删除文件、发送网络请求）：触发 L2 安全确认，等待用户批准。

**关键能力**：
- **沙箱隔离**：所有执行在沙箱（Docker 容器或临时目录）中进行，交付前不影响用户本地环境。
- **异步并行**：如果任务 DAG 中有并行节点，Agent 会同时发起多个工具调用。

---

### 2.3 👁️ Observe（观察）

**过程**：
1. **结果接收**：Agent 拿到工具的执行结果（成功、失败、具体数据、错误日志）。
2. **状态评估**：
   - **成功**：任务完成，发布 `TaskCompletedEvent`，返回结果。
   - **失败**：进入自修复循环。
3. **自修复循环**：
   - 分析错误日志（如构建错误、测试失败）。
   - 生成修复补丁（Patch）。
   - 回到第 1 步（Think），带着新的上下文再次思考、决策。
   - **重试限制**：达到最大重试次数（如 3 次）后，发布 `TaskFailedEvent`，请求人工干预。

**关键能力**：
- **错误感知**：Agent 能区分“临时性错误”（如网络超时，可重试）和“永久性错误”（如语法错误，需修改代码）。
- **交互式回溯**：用户可以在任意步骤暂停，查看中间产物，甚至可以回退到某个决策点，修改约束后重新执行。

## 🚀 三、进化层：从“工具”到“思想熔炉”的跨越

这是我们探索的终点，也是 Cortex 的终极形态。Agent 的设计不再是如何完成任务，而是**如何成为你的一部分**。

### 3.1 思想的延伸：意图预测（Thought Mirror）

**现状**：被动等待指令。  
**未来**：主动预测意图。

**实现方式**：
- **行为模式分析**：通过长期观察你的操作序列（如“打开文件 A → 搜索函数 B → 修改函数 C”），MetaAgent 学习你的工作流。
- **预判与建议**：在你刚打开一个文件时，Agent 可能就预判了你的下一步操作（如“你可能想运行测试”），并主动提供建议按钮。
- **思维可视化**：MetaAgent 的思维过程通过 `ThinkingStream` 实时展示，让你了解决策背后的逻辑（“我之所以推荐这个方案，是因为你上次说过喜欢简洁的代码”）。

---

### 3.2 技能的拓展：自动捕捉与复用（Skill Memory）

**现状**：每次都要重新教。  
**未来**：一次学会，终身复用。

**实现方式**：
- **成功经验提取**：当 Agent 成功解决一个复杂问题（如“配置 CI/CD 流水线”）后，`SkillExtractor` 会自动捕捉操作序列，提炼成一个可复用的“技能模板”。
- **技能存储**：技能模板存储到 `SkillMemory`（SQLite + 向量索引），包含前置条件、执行步骤、预期结果。
- **自动匹配**：下次遇到类似问题，Agent 会从技能库中检索匹配的技能，并在征得你同意后，自动帮你完成。
- **社区共享**：你可以将技能导出，分享给团队或社区，形成“技能市场”。

---

### 3.3 认知的突破：主动引入新知识（Cognitive Breakthrough）

**现状**：知识边界固定。  
**未来**：主动打破边界。

**实现方式**：
- **前沿监测**：`KnowledgeUpdater` 定期爬取技术文章、文档、最佳实践摘要，更新到 `CognitiveKB`（认知知识库）。
- **关联推荐**：当你在做某个项目时，Agent 会发现能提升你当前项目效率的新技术或新范式（如“我注意到 WebAssembly 在边缘计算中的新应用，这可能比传统的 Serverless 更适合你的博客部署场景”）。
- **范式转移提醒**：当行业出现重大技术变革时，Agent 会主动推送“认知突破”建议，帮你打破知识的边界。

---

### 3.4 信任分级模型：安全与效率的平衡

整个自主性被一个**信任分级模型**安全地约束着。从“新手”到“专家”，你对 Agent 的信任是逐步建立的，它的自主权限也随之逐步放开。

| 信任等级 | 用户特征 | Agent 自主权限 | 确认策略 |
|---------|---------|---------------|---------|
| **L1: 新手** | 初次使用，不熟悉 Agent 能力 | 仅只读操作（读取文件、搜索） | 所有写操作需 L2 确认 |
| **L2: 学徒** | 使用一段时间，建立初步信任 | 低风险写操作（创建文件、重命名） | 高风险操作需 L2 确认 |
| **L3: 专家** | 长期使用，高度信任 | 中风险操作（提交 Git、安装依赖） | 破坏性操作需 L2 确认 |
| **L4: 伙伴** | 完全信任，授权全自动 | 所有操作（包括部署、删除） | 仅 L1 蓝图确认 + L3 交付确认 |

**关键原则**：
> **用户始终是最终决策者**。即使在高信任等级下，L1（蓝图确认）和 L3（交付确认）仍然是强制的。
