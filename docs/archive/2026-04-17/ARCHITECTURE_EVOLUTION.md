## 一、核心设计理念

### 记忆系统是唯一的真理来源（Single Source of Truth）

> **所有功能模块都是记忆系统的“端口”——它们读取记忆、写入记忆、响应记忆的变化。**

本架构的三大核心原则：

1. **记忆主动驱动**：记忆系统不是被动存储，而是主动驱动整个系统行为
2. **功能即端口**：代码解释、提交生成等功能本质上是记忆的消费者和生产者
3. **事件围绕记忆**：事件总线以记忆系统为核心，所有重要事件都围绕记忆操作展开

### 设计目标

- ✅ **内核稳定**：记忆蒸馏、安全授权等核心能力不随功能增减而变化
- ✅ **插件热插拔**：用户可启用/禁用功能，甚至编写自己的Skill
- ✅ **技术栈可替换**：通过适配器，LLM、数据库、Git实现都可以随时更换
- ✅ **事件解耦**：模块间无直接依赖，易于测试和维护
- ✅ **半开放半封闭**：核心层封闭，插件层开放

## 三、核心设计原则

| 原则 | 说明 |
|------|------|
| **内核最小化** | 核心层只包含记忆系统、安全授权、事件总线、会话管理，其他所有功能均为插件 |
| **端口依赖** | 核心层只依赖端口（接口），不依赖具体适配器 |
| **事件驱动** | 插件之间、插件与核心之间通过事件总线通信，不直接调用 |
| **半封闭半开放** | 核心层封闭（不允许随意修改），插件层开放（用户可编写、可扩展） |
| **本地优先** | 所有适配器优先实现本地版本（如SQLite、Ollama），云端为可选 |

## 四、事件总线集成

### 4.1 现有EventBus设计

项目中已有事件总线机制（通过AuditLogger和tsyringe容器），需确保：

1. **模块间通信通过事件**：
   ```typescript
   // 示例：索引构建完成事件
   this.eventBus.emit('index.built', { count: totalDocs });
   
   // 订阅者
   this.eventBus.on('index.built', (data) => {
     console.log(`Indexed ${data.count} memories`);
   });
   ```

2. **保持AuditLogger审计链路**：
   - 所有关键操作仍通过AuditLogger记录
   - FeedbackRecorder继承此模式

### 4.2 事件定义

建议定义以下领域事件：

| 事件名 | 触发时机 |  payload |
|--------|---------|----------|
| `memory.recorded` | 记忆记录成功 | `{ id, taskType, memoryTier }` |
| `index.built` | 索引构建完成 | `{ count, duration }` |
| `search.executed` | 搜索执行 | `{ query, resultCount, duration }` |
| `feedback.recorded` | 用户反馈 | `{ query, clickedId }` |
| `memory.migrated` | 记忆迁移 | `{ count, fromTier, toTier }` |

## 六、技术债务与风险

### 已知问题

1. **ExpertSelector API不匹配**
   - FeedbackRecorder期望`updateFeedback()`方法
   - 实际ExpertSelector需要`IntentVector`和`RetrievalWeights`
   - **解决**：简化FeedbackRecorder，仅记录审计日志

2. **RetrievalWeights缺少v字段**
   - SearchEngine返回的权重对象缺少vector权重
   - **解决**：已添加v=0.1预留字段

3. **向后兼容性**
   - EpisodicMemory的公共API必须保持稳定
   - **解决**：通过适配器模式封装子模块调用

### 风险评估
